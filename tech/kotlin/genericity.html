# 泛型

我们都知道面向对象的编程语言具有三大特性：封装，继承和多态的特性（抽烟喝酒烫头）。它允许将一个基类的引用指向子类的实例对象：

SubClass sub = SubClass()
SuperClass superClass=sub

继承和多态可以让程序设计变得更为抽象和通用。然而，当我们讨论泛型的时候，由于类型擦除的原因，通常意义上的继承失效了。

泛型机制是通过JVM的类型擦除实现的，无论kotlin还是java，泛型类的实例对象在运行时并不会保存该泛型类的参数类型信息。

例如，我们创建了一个List<String> 实例对象dataList来存储一组String对象。尽管编译器不会让我们向dataList里面添加其他类型的数据，但在程序运行阶段，该实例对象dataList只是一个List类型的对象，并不知道里面存储的具体对象类型。

由于在运行时JVM无法来判断所操作的参数类型，就有可能带来数据类型不一致的风险，例如：

fun addData(list:MutableList<Any>){
    list.add(22)
}
val strList = mutableListOf("b", "a")
addData(strList)// 无法编译通过

在进一步讨论之前，我们需要弄清楚几个特殊的概念

  • 类class和类型type
  • 子类subClass和子类型subtype
  • 型变Variance：协变covariance，逆变contravariance以及非型变关系

为了便于理解这几个拗口的名词，虽然我使用了业内通用的名词翻译，但是还是觉得没有很好地表现出英文单词的原意。

# 类Class 和 类型Type

无论是Java还是kotlin，通常情况下，我们不会去区分类与类型这两个概念的。类Class我们已经很熟悉了，它是我们为创建的某个对象所定义的模板，而一个变量的类型type是指这个变量本身可存储的值.

一个非常简单的例子就是,定义一个变量str var str:String? ，它的值可以是一个String类的实例对象，也可以是一个null对象。在我们的认知中，String是通常意义上的类class，而String? 是一种类型type。很明显，kotlin中一个类Class最少对应着两种类型。

对于集合也是一样，List是一个类class，而List<String>并不是一个class，而是一个类型type。

# 子类subclass 和 子类型subtype

基类和子类比较容易理解，它是相对于继承这个概念而言的：也就是说如果B是A的子类subclass，那么B需要继承A。 而要理解子类型subtype的概念则需要举一个具象化的例子：一个变量var str:String?可以保存两种类型的值——String类型和 null类型，我们可以把“abc”和null赋值给str。这里，我们认为String就是String？的子类型，因为可以使用String?的值的地方（可以是非空String值也可以是null值）都可以换成String非空的值。我们似乎可以得出两个结论：

  • 如果B继承了A，那么B是A的子类，也是A的子类型
  • A是A?的子类型

# 型变Variance与非型变Invariance

通常情况下，我们其实并不关心子类和子类型的概念和区别，而在我们讨论泛型的时候，它们的关系就显得十分重要了。

在上面讨论类型擦除的例子中，MutableList<Any> 与 MutableList<String> 两个泛型类二者之间并无关系，即便String是Any的子类。

同样，在Java中参数类型之间的关系也不会影响整个泛型类之间的关系，这种也被称为非型变性Invariance。

而我们接下来要聊的型变Variance关系，则描述了泛型类中参数类型存在继承关系时，影响整个泛型类之间的关系的场景：例如List<Any>和List<String>。

# 协变Covariance

在kotlin中，泛型接口List<E>接口是一种只读列表，创建后无法修改列表内容。如果B类型是A类型的子类型subtype，那么List<B>就是List<A>的子类型subtype。例如：

val strList:List<String>= listof("aaa","bbb")
val anyList:List<Any>= strList

显然，List<String>与List<Any>的关系与参数类型String和Any之间的关系一致，这种性质被称为协变性，显然，List<E>具有可协变性。

那么在Kotlin中如何为泛型声明协变性呢？查看List<E>的定义可以发现：

interface List<out E> {
  fun get(index: Int): E
}

关键字out表示List中的方法只能返回类型E，而不能将类型E作为入参（除了构造函数之外），从而限定了E的协变性。我们在为一个泛型类型声明协变性的时候，务必保证参数类型E相关的引用无法被外界所修改。

# 逆变Contravariance

逆变的性质恰好和协变的性质相反，我们换一种泛型类Comparator<T>来介绍逆变：

val anyCompare: Compare<Any> = object: Compare<Any> {
  override fun compare(first: Any, second: Any){
    return first.hashCode() - second.hashCode()
  }
}
val strCompare: Compare<String> = anyCompare

String 是 Any的子类型，然而Compare<Any>是Compare<String>的子类型，这样的性质被称为逆变性。而定义泛型类的逆变需要使用in关键字来约束T：

interface Compare<in T> {
  fun compare(first: T, second: T): Int
}

在Compare的所有方法中，参数类型T只能作为入参存在，不能作为返回值类型。

而如果在Java的泛型类中使用协变和逆变，需要使用泛型类的时候使用通配符?和extends以及super关键字。

可以对比发现kotlin的型变定义在泛型类中，而Java的型变性质则需要使用该泛型类的时候才定义。但是显然在使用时对泛型的型变性质进行约束才更灵活啊，kotlin说好的处处针对Java呢？ 其实kotlin也支持在使用时定义泛型的型变性质。

# 投影

之前我们提到过kotlin的可变列表接口MutableList<E>具有非型变性，因为通常情况下MutableList会对其中的元素E的值进行生产（out）与消费（in）操作，也就是说在该接口的定义中，元素E可能作为入参存在，也可能作为返回值。 如果我们使用时能够保证程序中只使用MutableList<E>中的生产者方法（E作为返回参数）亦或是消费者方法（E作为入参），则可以使用类型投影的方式在MutableList<E>变量声明的地方使用out或in注解泛型E：

val list: MutableList<out Number>

这时MutableList就具有了协变性，同时我们无法再使用list的add等方法了。

除了类型投影以外，如果我们实在无法确认泛型参数的类型，可以使用星型投影的方式来声明泛型参数：List<*>, 它表示列表中存放的数据类型目前无法确认，但一定是某种特定类型；而List<Any?>则表示List中可以存放任意类型的数据。
