# 尝一口 Class —— “我们不一样”

尽管Kotlin的Class声明在很多地方与Java（特别是Java8）十分相像，但也吸收并引入了很多其他编程语言的优点与特性，从设计思想和细节上有太多的“不一样”！

# 类的定义与声明

我们知道，一个类的构造函数主要用于在创建该类的实例对象时完成相应的初始化工作。在Java中，我们可以为一个类声明一个或者多个构造函数，而kotlin在此之上又将构造函数分为了主构造函数和从构造函数。

# 主构造函数

每个类只能有一个主构造函数，负责采用最常用最便捷的方式来初始化一个类的实例对象。

class Button constructor(text:String){
    val mText:String 
    init {
        mText=text
    }
}

主构造函数在类名后用constructor关键字来声明，由于其声明的位置所限制，所以实例对象初始化相关的代码需要放在init块中完成，一个类中可以包含多个init块，按定义的顺序依次执行。

对于上面这个例子，可以进一步简化代码量：如果主构造函数没有被注解或者可见性修饰符所修饰，则constructor关键字可以被省略掉。另外，对于成员变量的赋值工作，可以直接使用构造函数的参数直接赋值，而不必在init块中显式赋值：

class Button (text:String){
    val mText:String = text
}

当然还可以进一步简化：

class Button (val text:String)// 如果成员属性是常量，需要用val来修饰

如果我们不需要对定义的类做任何初始化工作，也就没有必要定义构造函数。这时，kotlin会自动生成一个默认的构造函数。另外，如果我们定义的类继承了某个基类，主构造函数也需要完成对基类的初始化工作，最常用的做法是调用基类的构造函数来完成:

open class View(val id:Int)

class Button constructor(id:Int ,val text:String,  enable:Boolean = false):View(id){}

# 从构造函数

通常情况下，主构造函数可以满足实例对象初始化的需求。然而对于那些需要采用多种方式来初始化一个实例对象的需求，通常可以采用两种方式来实现：主构造函数参数指定默认值和定义从构造函数。 每个类可以有多个从构造函数，以满足其他多种构造实例的需求。

由于Java不支持方法参数设置默认值，通常需要通过方法重载来实现相应的需求，因此需要定义多个构造方法来支持。而kotlin通过为主构造函数设置参数默认值的方法就可以满足大部分需求了，对于那些真的需要不同参数来实例化对象的需要，就需要通过定义从构造函数来满足了。 对于从构造函数，我们可以定义多个：

open class View{
    constructor(id:Int){
    }
    constructor(id:Int,name:String){
    }
}

然而每个从构造函数都必须调用主构造函数（默认生成的主构造函数会自动调用）来完成对象的初始化工作，因此，即便用从构造函数来初始化类的实例对象，init块中的内容仍然会执行。

还有一个原则就是，当对象初始化的时候，需要告诉编译器如何去初始化它的基类，也就是要调用基类的某个构造函数：

class Button :View{
    constructor(id: Int) :super(id){

    }
    constructor(id: Int, name: String) :super(id,name){

    }
}

# Interface的定义与实现

与Java 8中的Interface几乎一样，Kotlin的接口定义不仅可以包括抽象方法，也可以包括非抽象方法：

interface A{
   fun a() // 默认的abstract方法，需要被实现类实现
   fun callMe()= print("AAA") //非abstract方法，对比java8 default方法
}

interface B{
   fun b()= print("hei hei hei") //非abstract方法
   fun callMe()= print("BBB")
}

class C:A,B{
   override fun callMe() {// 必须提供callMe方法的实现
       super<B>.callMe()  // 调用B 的方法实现
       super<A>.callMe()
   }
   override fun a() {
   }
}
  • Interface中定义的方法默认为abstract的，因此open关键字可以被省略，实现类C中必须提供a()方法的实现;
  • 对于B中所提供的非abstract方法b()，C可以不用提供相应的实现。
  • 尽管callMe()同样是非abstract方法，但由于A,B两个接口中都声明了相同的方法，编译器对于C具体实现哪个接口的方法是懵逼的，因此需要在C中显式提供callMe()方法的实现。

Kotlin可以在接口内声明属性：

interface A{
    val str:String 
}

接口中虽然声明了对应的属性，但是接口本身并不会存储属性的值，也不会关心属性如何存储；这些工作都会交给实现该接口的类来完成.

# 请为Class显式声明“可继承”

在Java中，显式地使用final关键字来修饰类或者其中定义的方法，表示该类不可以被其他类继承或者该方法不可以被子类覆写。

《Effective Java》一书中也提到了Java的这个设计上的缺点很可能会导致子类并没有按照基类所期望的方式和规则来完成继承，所以提出了“如果你不知道所定义的类是否会被继承的情况下，请使用final关键字让其无法被继承”的观点，以避免子类对于基类的随意继承。

显然，Kotlin的设计者赞同了这样的设计理念，希望开发人员在设计一个基类的同时制定好继承规则：

  • 如果Kotlin的Class不被显式地用open关键字修饰，那么它无法作为基类被继承；
  • 基类中只有那些被open关键字修饰的方法才能在子类中被覆写；
open class C:A,B{
    override fun callMe() {// 必须提供callMe方法的实现
        super<B>.callMe()
        super<A>.callMe()
    }
    open fun c(){}// 可以被子类覆写
    fun c1(){} // 不可被子类覆写
    override fun a() {// 可以被子类覆写
    }
    final override fun b() {// 不可以被子类覆写
        super.b()
    }
}

在Java中，子类如果覆写基类的方法，需要使用@override注解改方法；而在kotlin中则需要用override关键字来修饰。被override所修饰的方法默认情况下也是可以被子类所覆写的，除非像上面b()方法那样被加上了final关键字。

kotlin抽象类的设计几乎和Java一样，类及其中的抽象方法都需要用abstract来修饰,无法直接创建该类的实例等等。在kotlin中，abstract方法自然是可以被覆写和实现的，而抽象类中未被abstract或open修饰的方法自带final属性，无法被覆写。

# 可见性修饰符

为了满足程序的可见性与封装性，kotlin也采用了和java大致相同的可见性修饰符：private, protected以及public，不同的地方主要有两点：

  • 包可见性 Java默认情况下使用default对所修饰对象进行了“package”可见性控制，而在kotlin中，虽然也有“package”的概念，但是仅仅只用于代码命名空间的声明和划分，跟可见性没太大关系。kotlin实际上默认为public可见。

在kotlin中，设计者们提出了“module”的概念，也就是一些.kt文件作为一个集合一起被编译。而如果限定该集合内部的可见性，则需要使用kotlin所提供的internal关键字来修饰了。这样的设计方式提高了模块的封装性，降低了程序的耦合度。

  • protected可见范围缩小 在Java中，如果一个对象被声明为protected，那么我们可在同一“package”下随便访问该对象；而kotlin则对此做了限制：如果一个对象被声明为protected，那它只能在其本身和子类中进行访问。

# 内部类和嵌套类

和Java一样，kotlin也支持在一个类A的结构中定义另一个类B，这样不仅可以让代码的关联性更强便于管理，同时也提升了程序的封装性，这就是我们在Java中所熟知的内部类。

在Java中，如果内部类B没有被static关键字所修饰，也即B是一个非静态内部类，那么B的实例对象本身会持有外部类A的引用。

对应于Java的静态内部类，Kotlin引入了嵌套类的概念（Nested Class），嵌套类对象不会持有外部类的引用。

当然，如果我们真的需要内部类对象持有外部类的引用，则可以在声明内部类的时候使用inner关键字：

class Outer {
    inner class Inner {
    fun getOuterReference(): Outer = this@Outer 
    }
}

# Sealed类

Sealed类是kotlin关于封装内部类的一种特性：如果一个基类被sealed修饰，那么它所有直接继承它的子类都要作为其内部嵌套类来实现。

而sealed类经常与when语句来结合使用，如果所有的子类都囊括在了when语句中，那么就无需为when语句指定未匹配条件的默认处理逻辑了。

另外，sealed不能用来修饰一个接口，否则的话，如果Java类实现了该接口，就相当于密封性被打破了：直接子类在sealed类外部进行了定义。

# 通过使用object来定义类和实例对象

kotlin中提供了关键字object，可以用来定义一个类的同时创建该类的对象。这种操作听起来很像Java中的匿名内部类，但是object关键字可以应用在更多的场景中。

# 对象表达式

对象表达式其实就是上面所提到过的，利用object关键字来实现创建类似Java匿名内部类的做法。引用kotlin官网的例子：

window.addMouseListener(object : MouseAdapter() {
    override fun mouseClicked(e: MouseEvent) {
        // ……
    }

    override fun mouseEntered(e: MouseEvent) {
        // ……
    }
})

与Java的匿名内部类相比，通过object声明对象表达式更加方便：

  • object匿名对象可以实现多个接口
  • 在匿名对象中可以访问对象外部的变量，不需要强制改变量为final

# 创建单例

除了对象表达式，我们还可以通过object关键字来声明对象：

object Aa {
    fun a(str: String) {
    }
    val aList: List<String>
}

与对象表达式不同，这里object直接创建的对象并不是表达式，而像一个变量。在该对象中可以包含类的成员属性，方法，init块等等。然而object声明的对象会直接被实例化出来，无需通过调用构造函数。

整个对象的创建会在第一次调用的时候执行，其过程是线程安全的，在完成类定义的同时，也创建了该类的一个单例对象。而通过对象名称即可访问其中的方法：

Aa.a("a")

# 伴生对象

所谓伴生对象和Java中的静态成员变量非常像，但却是真正的实例对象，在该类第一次加载时创建，通过companion来声明：

class A{
    companion object Creator{
        fun create(){
        }
    }
}

在类中所定义的伴生对象不仅可以有对应的名称，实现接口，还可以实现函数和属性的扩展等等。
