# The Outputs of Webpack

在Webpack中，每个文件都被当作模块(module)来看待， 根据模块之间的依赖关系， Webpack从入口文件开始， 遍历并梳理各个模块之间的依赖关系，经过合并，打包等一系列操作，最终输出浏览器能够执行的文件。

基于 Webpack v4.2.9 ，本文从构建产物的角度对Webpack进行一系列分析，从而帮助我们来了解Webpack 对文件(模块)做了哪些操作。

# I. 准备工作

首先准备一下实验环境， 在工程中安装webpack以及webpack-cli：

  "devDependencies": {
    "webpack": "4.29.0",
    "webpack-cli": "3.2.1"
  }

为了分析构建结果，创建两个有依赖关系的module：

  • index.js 入口文件， 依赖了giraffe.js：
let giraffe=require('./giraffe')
function sayHi(){
  console.log("hi!")
  giraffe();
}
  • giraffe.js ：
function giraffeHi(){
  console.log("I'm Giraffe..")
}

module.export=giraffeHi;

# II. Output Analysis

执行构建命令 webpack index.js --mode=none 后， 可以在dist/ 目录中获得最终的构建结果main.js :

从输出的log信息可以看到， webpack编译了三个module，除了提供的两个文件之外，还有(webpack)/buildin/module.js

先来看一下构建结果main.js的内容，它整体上是一个自执行函数，被称为webpackBootstrap

先抛开红色部分函数体的具体内容，函数的入参是一个数组，数组元素与上面被处理的module一一对应。 可以看到， 每个module的具体内容被封装进了一个模块函数中：

(function(module, exports, __webpack_require__) {
  //module contents
}),

所以webpackBootstrap函数入参的数组中包含了三个模块函数，让我们先记住这个概念，随后的调用中会大量用到。

接下来看一下webpackBootstrap函数中做了哪些工作。 函数整体结构也很简单，分别定义了一个cache和一个require函数，以及围绕他们所定义的一些对象，我们逐步分析：

# 1. Module Cache

/******/ 	// The module cache
/******/ 	var installedModules = {};

webpackBootstrap 首先创建了一个module cache用来缓存已经加载过的module，数据结构如下图所示：

  • Id 标识module
  • loaded 标识是否已经加载过
  • module的export对象

# 2. Require Function

除了module cache， webpackBootstrap还定义了require函数，用来根据module Id获取 该module的exports对象:

function __webpack_require__(moduleId) {

	// 如果加载过该module则直接从缓存中取出并返回
	if(installedModules[moduleId]) {
		return installedModules[moduleId].exports;
	}
	// 加载新module前，创建一个新的缓存对象
	var module = installedModules[moduleId] = {
		i: moduleId,
		l: false,
		exports: {}
	};

	// modules是webpackBootstrap的入参数组，立即执行封装了该module的模块函数
	modules[moduleId].call(module.exports, module, module.exports, __webpack_require__);

	// 标识模块加载完毕
	module.l = true;

	// 返回module的exports
	return module.exports;
}

接下来webpackBootstrap为声明的require函数赋上了各种属性和方法：

  • 入参数组 webpack_require.m = modules

  • 模块缓存 webpack_require.c = installedModules

  • d函数可以用于为exports对象中指定的属性定义相应的getter函数

// define getter function for harmony exports
__webpack_require__.d = function(exports, name, getter) {
	if(!__webpack_require__.o(exports, name)) {
		Object.defineProperty(exports, name, { enumerable: true, get: getter });
	}
};
  • o函数用来判断其实就是Object.prototype.hasOwnProperty方法的封装
__webpack_require__.o = function(object, property) { 
  return Object.prototype.hasOwnProperty.call(object, property); 
};

# 3. Execution

随着require函数初始化完毕，webpackBootstrap也执行完毕，调用require函数开始加载入口文件并返回其exports

// Load entry module and return exports
return __webpack_require__(__webpack_require__.s = 0);

以实验中的代码为例， 执行入口模块的模块函数：


 






function(module, exports, __webpack_require__) {
  let giraffe=__webpack_require__(1)
  function sayHi(){
    console.log("hi!")
    giraffe();
  }
}

入口模块index依赖了giraffe模块，在定义sayHi函数之前，首先需要加载giraffe函数。

前面一样，__webpack_require__(1)的调用会执行giraffe模块的模块函数：



 








function(module, exports, __webpack_require__) {
  (
    function(module) {// ①
      function giraffeHi(){
        console.log("I'm Giraffe..")
      }
      module.export=giraffeHi;
    }.call(this, __webpack_require__(2)(module))
  )
}

这里的调用层级比较复杂，我们可以一层一层地解析。 该模块函数中定义并立即执行了函数 ①，入参部分首先调用了 __webpack_require__(2)加载了最后一个模块，并执行其模块函数：


 























function(module, exports) {
  module.exports = function(module) {// ②
    if (!module.webpackPolyfill) {
      module.deprecate = function() {};
      module.paths = [];
      // module.parent = undefined by default
      if (!module.children) module.children = [];
      Object.defineProperty(module, "loaded", {
        enumerable: true,
        get: function() {
          return module.l;
        }
      });
      Object.defineProperty(module, "id", {
        enumerable: true,
        get: function() {
          return module.i;
        }
      });
      module.webpackPolyfill = 1;
    }
    return module;
  };
}

该模块函数将module2的exports对象指向了一个函数 ② ， 它的作用是负责构造传入的module对象。 在上面函数 ① 的执行中调用了函数 ②：



 








function(module, exports, __webpack_require__) {
  (
    function(module) {// ①
      function giraffeHi(){
        console.log("I'm Giraffe..")
      }
      module.export=giraffeHi;
    }.call(this, __webpack_require__(2)(module))
  )
}

这里调用__webpack_require__(2)(module) 传入的module对象其实是module1， 也就是用函数 ② 来构造了module1。

随后在函数 ① 的调用中， module1的exports对象被初始化为giraffeHi函数， 并通过

let giraffe=__webpack_require__(1)的调用返回给入口文件的调用。 至此，整个层层调用过程完毕，尽管比较繁琐， 但整体流程还是比较清晰的。

# III. 总结

通过对Webpack构建产物的分析和梳理，我们可以发现， Webpack采用了缓存对象来维护模块的加载与使用，而对于模块的依赖关系则通过require函数来完成。 从入口文件开始，递归引用其他模块的同时， 只关心和缓存所依赖模块对外的接口（能力），而具体实现仍然被封装在模块内部不对外暴露。
