# 安全漏洞与问题处理

Webview为业务发展提供了强大的动态化和引流能力， 同时也为业务开发提供了跟多灵活的选择。 但在其使用过程中也有很多安全漏洞和问题被暴露出来，例如任意代码执行，跨域问题等等。

同时，伴随着Webview与内核的发展与迭代，高版本中修复的的Bug和漏洞，仍需要我们在低版本中进行兼容和处理。

# JavascriptInterface 注入对象漏洞

  • 在Web端调用客户端方法 中所提到的， Webview 注册JavascriptInterface对象，可以实现Web端对于客户端方法的调用。 在Android 4.2以下版本中，通过Java的反射机制，攻击者可以获取到系统类对象为所欲为。

在Android 4.2 以上版本中， 为了解决该问题， Webview规定只有被@JavascriptInterface注解过的方法才能被Web端调用到。

如果不在4.2以下系统中注册JavascriptInterface对象， 是否就可以规避该漏洞带来的风险呢？ 答案是NO.

即便我们不主动向Webview中注册JavascriptInterface对象， 但由于Webview自身默认注册的对象也会被攻击者利用，因此我们需要移除掉相关JavascriptInterface对象：

removeJavascriptInterface("searchBoxJavaBridge_");
removeJavascriptInterface( "accessibility");
removeJavascriptInterface("accessibilityTraversal");

# file协议的安全性

在实际应用过程中，我们通常需要Webview能够访问或加载本地文件的功能， 这就需要WebSettings允许开启文件访问功能setAllowFileAccess(true)。

功能增强的同时，也带来了一定的安全风险， 需要进行一系列的拦截和处理措施。

除了setAllowFileAccess， WebSettings中还有两个设置：

  • setAllowFileAccessFromFileURLs (boolean flag) 设置是否允许以 file:// URL的方式加载的Javascript访问本地其他文件内容。

  • setAllowUniversalAccessFromFileURLs (boolean flag) 设置是否允许以 file:// URL的方式加载的Javascript访问其他源的数据。

当这两种配置开启的时候， Webview加载的本地Javascript可以访问本地文件和其他域的内容。 为了避免被恶意攻击， 建议每次加载url之前， 对相应的URL进行校验。 如果是可信任的URL， 则启用设置。

除此之外，在加载URL之前， 建议对URL进行校验， 不去加载有安全隐患的URL。
